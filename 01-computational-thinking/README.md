## 0. 컴퓨팅 사고(Computational Thinking)란?

문제를 *컴퓨터(혹은 AI)가 다룰 수 있는 형태*로 바꿔 사고하는 방법. 이 문서는 아래 **4가지 렌즈**로 내 업무를 들여다본다.

| 요소 | 뜻 | 이 문서에서의 질문 |
|------|----|--------------------|
| 🧩 **분해**(Decomposition) | 큰 문제를 작은 단위로 쪼개기 | "이 업무는 어떤 단계들로 이뤄지는가?" |
| 🔁 **패턴 인식**(Pattern Recognition) | 반복·규칙 찾기 | "매주 똑같이 반복되는 부분은?" |
| 🎯 **추상화**(Abstraction) | 핵심만 남기고 곁가지 제거 | "본질적인 입력·규칙·출력은?" |
| ⚙️ **알고리즘**(Algorithm) | 순서 있는 절차로 정리 | "누가 봐도 따라 할 수 있는 절차는?" |

## 1. 내 도메인 소개 & AI 서비스 관점 정리

### 1-1. 하는 일

서비스 기획/PM/운영으로서, **서비스가 잘 굴러가는지 관찰하고 → 문제를 발견해 → 개선을 굴리는** 사이클을 담당한다.

- 서비스 지표(가입·이탈·전환 등) 모니터링
- 고객의 소리(VOC)·문의 취합 및 대응 우선순위 결정
- 기능 개선/신규 기획 정리, 이해관계자(개발·디자인·경영진) 커뮤니케이션
- **주간/월간 운영 리포트 작성 및 공유**

### 1-2. 핵심 역량

`문제 정의` · `우선순위 판단` · `데이터 해석` · `커뮤니케이션/문서화`

### 1-3. 반복 업무를 "AI 서비스 관점"으로 다시 보기

> AI 서비스 관점 = 모든 업무를 **입력 → 처리(규칙) → 출력** 의 파이프라인으로 바라보는 것.

| 반복 업무 | 입력(Input) | 처리(규칙) | 출력(Output) | 반복 주기 |
|-----------|-------------|-----------|--------------|-----------|
| 운영 리포트 작성 | 지표 데이터, VOC 로그 | 집계·비교·요약 | 리포트 문서 | 주 1회 |
| VOC 분류 | 고객 문의 텍스트 | 카테고리 규칙 | 태그된 이슈 목록 | 매일 |
| 회의록 정리 | 회의 녹취/메모 | 요약·액션아이템 추출 | 정리된 회의록 | 회의마다 |

> ✍️ **이렇게 바꿔보세요:** 위 표를 *본인이 실제로 매주/매일 하는 업무*로 교체하면 됩니다.

이 중 **가장 시간이 많이 들고 규칙성이 강한** `운영 리포트 작성`을 대표 업무로 골라 분해한다.

## 2. 대표 업무 분해 (Decomposition) 🧩

**"주간 서비스 운영 리포트 작성"** 을 3단계 → 하위 작업 → 세부 액션으로 쪼갠다.

```
주간 운영 리포트 작성
├── A. 데이터 수집
│   ├── A1. 지표 대시보드에서 주요 수치 추출 (가입/이탈/전환/DAU)
│   ├── A2. 전주 대비 증감 계산
│   └── A3. VOC/문의 로그 내려받기
├── B. 분석·해석
│   ├── B1. 이상치·급변 지표 탐지
│   ├── B2. VOC를 카테고리별로 분류·집계
│   ├── B3. "왜 변했는가" 원인 가설 메모
│   └── B4. 이번 주 핵심 이슈 Top 3 선정
└── C. 문서화·공유
    ├── C1. 리포트 템플릿에 수치/그래프 채우기
    ├── C2. 요약(Executive Summary) 작성
    ├── C3. 액션 아이템·담당자 정리
    └── C4. 슬랙/메일로 공유 및 회의 안건화
```

| 단계 | 하위 작업 수 | 대략 소요 | 난이도 |
|------|:-----------:|:---------:|:------:|
| A. 데이터 수집 | 3 | 40분 | 낮음(단순 반복) |
| B. 분석·해석 | 4 | 60분 | 높음(판단 필요) |
| C. 문서화·공유 | 4 | 50분 | 중간 |

## 3. 패턴 인식 (Pattern Recognition) 🔁

분해해 놓고 보니 **매주 똑같이 반복되는 규칙**이 보인다.

- **완전 반복(규칙 고정):** A1·A2·A3, C1 — 매주 같은 소스에서 같은 방식으로 수치를 뽑고 같은 템플릿을 채운다.
- **반복이지만 판단 개입:** B1·B2·B4 — 절차는 같지만 "무엇이 이상한가/중요한가"라는 해석이 필요.
- **매번 새로움:** B3(원인 가설), C2(요약 메시지) — 맥락 의존적.

### 병목 & 실수 잦은 지점

- ⏱️ **병목:** A단계(데이터 수집)가 단순 반복인데도 전체 시간의 30%를 잡아먹음.
- ⚠️ **실수:** 수기 복사·붙여넣기 과정에서 숫자 오기입, 전주 대비 계산 실수.
- 🔁 **비효율:** 매주 "지난주엔 어떻게 썼더라" 하고 과거 리포트를 다시 찾아봄.

> **인사이트:** 반복+규칙 고정 구간(A, C1)은 **자동화 후보 1순위**, 판단 구간(B)은 **AI 보조(초안 제시) 후보**다.

## 4. 추상화 (Abstraction) 🎯

곁가지(어떤 대시보드 UI인지, 색깔은 뭔지 등)를 걷어내고 **본질만** 남긴다.

**"주간 운영 리포트"의 본질 = 아래 스키마를 채우는 일**

```yaml
입력(Input):
  - metrics: 주요 지표 값 + 전주 값
  - voc_logs: 고객 문의 텍스트 목록
규칙(Rule):
  - 증감률 = (이번주 - 전주) / 전주
  - VOC는 사전 정의된 카테고리로 분류
  - |증감률| >= 임계치 → "주목 이슈"로 승격
출력(Output):
  - summary: 3줄 요약
  - highlights: 핵심 이슈 Top 3
  - actions: 액션 아이템 목록(담당자 포함)
```

이렇게 추상화하면, "리포트 작성"이라는 막연한 일이 **정해진 입력을 받아 정해진 출력을 만드는 함수**로 바뀐다. → 곧 자동화·에이전트화가 가능하다는 신호.

## 5. 알고리즘 / 절차화 (Algorithm Design) ⚙️

누가 봐도 따라 할 수 있는 절차로 정리하고, 각 단계에 **자동화 수준**을 태깅한다.
(`🟢 자동 가능` / `🟡 AI 보조(사람 검수)` / `🔴 사람 판단 필수`)

```text
function 주간운영리포트():
    1. metrics ← 대시보드에서 지표 추출          # 🟢 자동
    2. for m in metrics: 증감률 계산             # 🟢 자동
    3. voc ← 문의 로그 수집                       # 🟢 자동
    4. voc_tagged ← 카테고리 분류(voc)            # 🟡 AI 보조
    5. issues ← |증감률| 큰 항목 + voc 급증 항목   # 🟡 AI 보조
    6. top3 ← issues 중 중요도 상위 3개 선정       # 🔴 사람 판단
    7. cause ← 각 이슈의 원인 가설 작성            # 🔴 사람 판단
    8. draft ← 템플릿에 값·요약 채우기            # 🟡 AI 보조
    9. 사람 검수 후 슬랙/메일 공유                 # 🔴 사람 판단
```

| 자동화 수준 | 해당 단계 | 비중 |
|-------------|-----------|:----:|
| 🟢 자동 가능 | 1, 2, 3 | ~35% |
| 🟡 AI 보조 | 4, 5, 8 | ~40% |
| 🔴 사람 판단 | 6, 7, 9 | ~25% |

→ **약 75%의 단계를 AI로 자동화하거나 초안화**할 수 있다는 결론.

## 6. AI 에이전트 관점 재구성 🤖

위 절차를 "사람이 지시하면 대신 수행하는 에이전트"로 다시 그리면:

```
[사용자] "이번 주 운영 리포트 만들어줘"
      │
      ▼
 ┌─────────────────────────────┐
 │      운영 리포트 에이전트       │
 │  (계획 → 도구 호출 → 종합)     │
 └─────────────────────────────┘
      │  필요한 도구(Tool)
      ├─ 📊 지표 조회 도구  (대시보드/DB API)
      ├─ 🗂️ VOC 분류 도구  (LLM 텍스트 분류)
      ├─ 📈 증감 계산 도구  (규칙 기반 연산)
      └─ 📝 문서 생성 도구  (템플릿 + 요약 LLM)
      │
      ▼
 리포트 초안 → 사람이 Top3·원인·최종 검수 → 공유
```

- **에이전트가 대신할 부분:** 데이터 수집·계산·분류·초안 작성(1~5, 8)
- **사람이 남길 부분:** 중요도 판단·원인 해석·최종 승인(6, 7, 9)
- 이것이 곧 아이펠 과정에서 배울 **Tool Calling / RAG / 멀티 스텝 에이전트**와 직접 연결된다.

## 7. 개인 문제 정의 초안 (과정 전체 프로젝트) 🎯

> 이번 과정 내내 붙잡고 발전시킬 **나의 문제**.

- **문제 정의 (한 문장):**
  *"서비스 기획자/운영자가 매주 1~2시간씩 쓰는 반복적 운영 리포트 작성을, 데이터 수집–분석–초안 작성까지 대신 처리하는 AI 에이전트가 없다."*

- **대상 사용자:** 지표·VOC를 주기적으로 리포트하는 소규모 팀의 기획자·PM·운영자

- **Pain Point:**
  - 단순 수집·복붙에 시간 낭비, 숫자 오기입 리스크
  - 매주 같은 포맷을 처음부터 다시 만드는 비효율
  - 정작 중요한 "해석·의사결정"에 쓸 시간이 부족

- **가설:**
  *리포트 업무의 반복 구간(수집·계산·분류·초안)을 에이전트가 처리하면, 소요 시간을 절반 이하로 줄이고 사람은 판단에 집중할 수 있다.*

- **해결 아이디어 (에이전트 형태):**
  지표 조회·VOC 분류·문서 생성 도구를 갖춘 **"주간 운영 리포트 에이전트"** — 사용자가 한 번 요청하면 초안까지 만들어 주고, 사람은 Top3·원인·최종 검수만 수행.

- **성공 지표:**
  - 리포트 작성 시간 **150분 → 60분 이하** (약 60% 절감)
  - 수치 오기입 **0건**
  - 사용자가 "초안을 그대로 또는 약간 수정해 쓴 비율" **80% 이상**

- **제약 · 리스크:**
  - 데이터 소스 접근 권한/보안
  - LLM의 수치·요약 신뢰성(환각) → 사람 검수 단계 필수
  - VOC 분류 정확도는 학습/프롬프트 개선 필요

> ✍️ **이렇게 바꿔보세요:** 위 수치(150분, 80% 등)와 대상 사용자를 *본인의 실제 상황*에 맞게 조정하세요.
